# Platform asset construction

## Universal source rule

Generate every target file independently from the highest-resolution canonical artwork. Never build 40px from 32px, 96px from 48px, or a mobile asset from an ICO frame. Treat crop, mask, flattening, and semantic optimization as target-specific transforms applied to the original source.

Apply semantic matte and core inheritance only to actual pixel sizes that need it, normally 48px and below. Render larger assets directly from the canonical source so small-pixel corrections cannot leak into high-resolution artwork.

## Windows

Reference: [Construct your Windows app's icon](https://learn.microsoft.com/zh-cn/windows/apps/design/iconography/app-icon-construction) (Microsoft Learn; source page dated 2022-06-02).

Windows looks for an exact pixel-size match first. If none exists, it selects the next larger asset and scales down. Use the built-in `windows-full` CLI preset:

```text
16, 20, 24, 30, 32, 36, 40, 48, 60, 64, 72, 80, 96, 128, 256
```

Microsoft's minimum set is 16/24/32/48/256. The full set covers title bar, tray, context menu, taskbar, search, Start, and Windows 11 display scaling from 100% through 400% with fewer fallback resizes.

For packaged AppList assets, Microsoft documents target sizes 16/20/24/30/32/36/40/48/60/64/72/80/96/256 and requires default, dark unplated, and light unplated variants. Missing unplated variants can cause an unwanted system backplate. Generate those files only when the packaging manifest consumes them; do not add unused copies to a Win32-only bundle.

Keep transparency unless the brand explicitly requires a plate. Provide separate light/dark artwork when contrast differs materially.

## macOS

Build the ICNS representations independently at 16/32/64/128/256/512/1024 actual pixels. Include standard and Retina representations in the ICNS container. Optimize only the truly tiny 16px and 32px raster representations; do not upscale them for Retina or larger entries.

## Linux

Generate committed PNG sizes directly from the canonical source. Include every size consumed by the bundle configuration. Keep alpha for rounded or non-rectangular icons.

## Android

Generate legacy launcher icons at 48/72/96/144/192px for mdpi through xxxhdpi. Generate adaptive foreground canvases at 108/162/216/324/432px directly from the canonical source and preserve the platform safe zone. Keep the adaptive background as a separate resource. Do not derive higher-density assets from mdpi.

## iOS and iPadOS

Read pixel sizes from `Contents.json` as points multiplied by scale. Generate each declared file directly from the canonical source. AppIcon PNGs must be opaque RGB without alpha; flatten onto the intended background after rendering. Let the OS apply the final corner mask.

## Release verification

- Confirm every file's exact dimensions and color mode.
- Decode ICO and ICNS containers and list their embedded representations.
- Compare regenerated bytes with committed assets through a check command.
- Inspect 16/20/24/29/30/32/36/40/44/48px at actual size and nearest-neighbor magnification.
- Confirm the largest output never exceeds the canonical source dimensions unless the crop decision explicitly accepts resampling from that same high-resolution source.
